# ADR-0001: 하네스는 repo-local로 직접 소유하고, ECC는 "시드"로만 참고한다

- 상태: Accepted
- 날짜: 2026-06-15

## 배경

이 프로젝트의 주목적은 **AI 에이전트 하네스 구축**이다(파일 업로드 서비스는 적용 도메인).
하네스 출발점으로 ECC(everything-claude-code, `affaan-m/ECC`) 플러그인을 검토했다.

ECC는 **프로젝트 단위 설치/활성화가 가능하다** (`.claude/settings.json`의 `enabledPlugins`,
`.claude/rules/` 로컬 복사). 따라서 "설치 범위"는 문제가 아니었다. 실제로 막힌 지점은 다음이다:

1. **커스터마이즈 불투명** — 플러그인의 에이전트·스킬·훅은 마켓플레이스 캐시
   (`~/.claude/plugins/cache/ecc/…`)에 들어 있어 사실상 읽기 전용이고, 하나만
   프로젝트에 맞게 바꾸려면 fork/shadow가 필요하다. "무엇을 어디까지 바꿔도 되는지"가 모호하다.
2. **에이전트 과다 + 통제 불가** — 설치 시 `~/.claude/agents/`에 60+개(네임스페이스 포함
   100+개)가 쏟아지고, 대부분 이 프로젝트와 무관하며(rust/harmonyos/healthcare…)
   선택적으로 솎아낼 손잡이가 없다.
3. **소유권** — 버전을 마켓플레이스가 소유하므로 업데이트가 로컬 수정을 덮을 수 있다.

규칙(rules)은 프로젝트로 복사해 고칠 수 있지만, 복사·수정하는 순간 그건 이미
"플러그인"이 아니라 **우리 파일**이다.

## 결정

ECC 플러그인을 **사용하지 않는다.** 대신:

- 하네스 자산(규칙·스킬·에이전트)을 **이 repo의 `.claude/`에 직접 두고 git으로 소유**한다.
- ECC의 내용은 **"시드(seed)"로만 가져와** 이 프로젝트에 맞게 정리한 뒤 우리 규칙으로 관리한다.
  (없는 공통 규칙/에이전트/스킬 참조 제거, 도메인 특화 항목 추가.)
- 필요한 것만, 필요할 때 추가한다. 기본값은 "비어 있고 통제된 상태".

### 이번에 적용한 것

- `rules/java/*` (coding-style·patterns·security·testing) — ECC `rules/java` 시드 후 정리.
- `skills/springboot-tdd`, `skills/springboot-patterns` — ECC 스킬 시드.
  - 선정 이유: 스택이 Java + Spring Boot이고, 작업 규칙이 **TDD 우선**이라 두 스킬이 직결된다.
    `quarkus-*`는 쓰지 않으므로 제외. `springboot-security`/`security-review`/`jpa-patterns`는
    해당 단계(보안·DB)에 도달할 때 추가한다.
  - 규칙(무엇을) ↔ 스킬(어떻게)의 보완 관계를 따른다. 스킬은 규칙 동작의 필수 의존이 아니다.

## 결과

**장점**
- 하네스 구성이 repo 안에 있어 `git log`로 전 과정이 추적·재현된다 (프로젝트 주제와 일치).
- 이 프로젝트에만 적용되며, 전역/다른 프로젝트에 영향 없다.
- 무엇이 켜졌는지 명시적이고, 100% 우리가 소유·수정한다.

**단점 / 비용**
- 업스트림 자동 업데이트가 없다 — 좋은 변경은 수동으로 다시 시드해야 한다.
- 시드한 자산은 죽은 참조/타 언어 예제를 직접 정리해야 한다.
- 유지 책임이 우리에게 있다 (그게 곧 학습 목표이기도 하다).
