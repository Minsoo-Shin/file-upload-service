# 대용량 파일 업로드 서비스 (Java Spring 클라이언트/서버)

> 이 프로젝트의 주목적은 **AI 하네스 구축**이며, 이 PRD는 하네스를 적용하는
> **도메인(업로드 처리 학습)** 의 요구사항이다. (ADR-0001)
> *요구사항(무엇/왜)만 다룬다. 구현(어떻게)은 Plan 모드로.*

## Problem
"파일 업로드"는 간단해 보이지만, 안정적으로 만들려면 스트리밍·검증·청크·재개·중복제거 등
마주쳐야 할 처리가 많다. 그런데 이 처리들은 **여기저기 흩어져 있고, 만들어가는 과정이
기록으로 남지 않아** 매번 다시 헤맨다. 특히 사내망에서 ~10GB 동영상을 multipart로 올릴 때
느리고 끊기는데, 재시도·멱등성에 대한 레퍼런스가 없었다. 이 처리들을 **쉬운 것부터 하나씩
직접 구현·기록하며 익히고, 재사용 가능한 레퍼런스**로 남길 필요가 있다.

## Evidence
- 실제 경험: 대용량 동영상(~10GB) multipart 업로드를 구축했으나 네트워크 병목으로 매우 느림.
- 사내 인프라망이라 네트워크가 불안정해 업로드가 끊김.
- 업로드 재시도·멱등성 처리 튜토리얼/레퍼런스를 찾지 못함.
- (validated — 직접 구축 경험 기반)

## Users
- **Primary**: 업로드 처리를 직접 구축·학습하는 개발자(본인) — 회사에서 재사용할 레퍼런스 제작.
- **Secondary**: GitHub로 만들어가는 과정을 따라오는 개발자.
- **Not for**: 비개발자 / 일반 업로드 서비스의 최종 사용자 (웹 UI 없음).

## Hypothesis
We believe **업로드 처리를 간단→스트리밍→청크→재개→중복제거 순으로 하나씩 구현·테스트·
기록**하면 will **각 기법을 익히고, 과정이 추적되는 재사용 가능한 레퍼런스가 된다** for
**사내 대용량 업로드를 다루는 개발자(본인)와 과정을 따라오는 사람들**.

We'll know we're right when:
- 각 단계가 **동작하는 기능 + 테스트 + devlog**로 누적된다.
- 처음 보는 사람이 **커밋 + devlog만으로 과정을 따라올 수 있다.**
- 단계가 쌓여 결국 "**끊겨도 이어 올리고, 중복은 즉시 완료**"되는 업로드에 도달한다(아래 북극성).

> **North Star (방향을 정해주는 시나리오)**: *불안정한 사내망에서 10GB 동영상을 올린다.*
> 이 시나리오가 마일스톤 순서를 정해준다 — 당장의 MVP 목표가 아니라, 단계가 향하는 종착지다.

## Success Metrics
주 지표는 **학습·기록의 누적**이다. 기술 목표(아래 북극성)는 *해당 단계의 완료기준(DoD)*으로 검증한다.

| Metric | Target | How measured |
|---|---|---|
| 단계 완결성 | 각 마일스톤이 테스트 통과 + devlog 작성으로 마감 | 마일스톤별 DoD 체크 |
| 과정 추적성 | 외부인이 커밋 + devlog만으로 흐름을 재현·이해 가능 | README/devlog 리뷰 |
| 북극성 도달 | 재개·중복제거까지 단계가 누적되어 동작 | M5·M6 통합 테스트 |

**북극성 기술 목표 (해당 단계 DoD로 검증):**
- 메모리 안전(M2): 10GB 업로드 중 서버 힙이 파일 크기에 비례하지 않음(상수 수준)
- 재개 효율(M5): 끊김 후 재개 시 이미 보낸 데이터 재전송 ≈ 0 (미완료 청크 1개 이내)
- 중복(M6): 동일 콘텐츠 재업로드 시 본문 전송 0바이트, 즉시 완료

## Scope
**MVP** — 단일 파일 업로드가 end-to-end로 동작 (클라이언트 전송 → 서버 수신·저장 → 결과 응답).
가설("간단부터 하나씩")을 **첫 단계부터** 검증하는 출발점.

**Out of scope**
- 웹 UI — 클라이언트는 Spring 기반, UI 없음
- 인증/권한, 멀티테넌시 — 핵심 업로드 처리 학습에 집중
- 동영상 트랜스코딩 등 후처리
