# file-upload-service

**AI 에이전트 하네스를 직접 구축하고, 그 하네스로 파일 업로드 서비스를
단계적으로 만들어가는 프로젝트입니다.**

주된 목표는 *애플리케이션*이 아니라 **하네스**입니다 — Claude Code 위에
프로젝트 전용 스킬·규칙·워크플로를 직접 설계하고, 그것이 실제 기능 개발에서
어떻게 작동하는지 검증합니다. 파일 업로드 서비스는 그 하네스를 적용하는
**실전 도메인**입니다.

## 두 축

### 1. AI 하네스 구축 (주)

- 글로벌/서드파티 플러그인을 배제하고, **이 repo의 `.claude/`에 처음부터 직접** 쌓는다.
- 반복/절차가 긴 작업만 스킬로 만들고, 컨벤션은 가벼운 규칙으로 둔다.
- 하네스 구성은 [`.claude/`](.claude/)에서 볼 수 있다.

| 구성 | 위치 | 내용 |
|---|---|---|
| 규칙 | [`.claude/CLAUDE.md`](.claude/CLAUDE.md) | 프로젝트 규칙 + 커밋 컨벤션 |
| 스킬 | [`.claude/skills/`](.claude/skills/) | `create-git-repo` (git 초기화 + GitHub 원격) |

> 하네스는 기능을 만들어가며 **필요할 때마다 늘려갑니다.** 무엇을/왜 추가했는지는
> `docs/devlog/`에 기록합니다.

### 2. 적용 대상 — 파일 업로드 서비스

하네스를 실제로 굴려볼 도메인. "파일 업로드"는 간단해 보여도 제대로 만들려면
스트리밍·검증·청크·재개·중복제거 같은 처리가 필요해서, 단계를 쌓기에 좋은 소재입니다.

단일 스택: **Java(Spring) 서버 + Picocli CLI**.

#### 적용 로드맵 (단계별 누적)

| 단계 | 기능 | 다루는 처리 |
|---|---|---|
| M1 | 단일 파일 업로드 → 저장 | Spring 파일 수신(`MultipartFile`) |
| M2 | 스트리밍 저장 | 메모리 통째 적재 방지(OOM), `InputStream` |
| M3 | 검증 & 보안 | 크기 상한, MIME/매직넘버, 파일명 sanitize, path traversal |
| M4 | 청크 업로드 | 청크 분할·병합, 업로드 세션 API |
| M5 | 재개 가능(resumable) | 세션 상태 영속화, 누락 청크, idempotency |
| M6 | 중복제거(dedup) | SHA-256 콘텐츠 해시 인덱스 |
| M7 | 진행률 & 병렬 (CLI) | 클라이언트 동시성, 재시도, 진행바 |
| M8 | 스토리지 추상화 | 로컬 FS → S3 호환 (포트/어댑터) |
